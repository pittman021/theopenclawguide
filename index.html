Module 1

// WHAT IS OPENCLAW?

OpenClaw is an open-source AI assistant framework that runs locally on your machine. Think of it as having a digital chief of staff that:

  • Lives in your messaging apps (Telegram, Slack, WhatsApp)
  • Can access your files, calendar, and data
  • Executes tasks on your behalf
  • Remembers context across conversations

It's not a chatbot. It's an autonomous agent that works when you don't.

What OpenClaw Is NOT

  • Not a SaaS subscription you rent
  • Not a black box you can't inspect
  • Not just for developers

What OpenClaw IS

  • Open source (you own it)
  • Runs locally (your data stays yours)
  • Customizable to YOUR workflows
  • Free to run (just pay for API tokens)

// MODULE 1: FOUNDATION

Goal: Get OpenClaw running + your first working agent

Step 1: Installation & Setup

OpenClaw runs on Mac, Linux, or Raspberry Pi. Here's the quickstart:

# Clone the repo
git clone https://github.com/openclaw/openclaw.git
cd openclaw

# Install dependencies
npm install

# Run the setup
npm run setup

Step 2: Understanding Agents vs Sub-Agents

Agents = Independent workers with their own personality and memory
Sub-Agents = Task-specific helpers spawned by your main agent

Think of it like:

  • Agent = Your employee
  • Sub-Agent = A contractor you hire for one specific job

Step 3: Your First Agent — Daily Briefing

This is the simplest starting point. Your agent sends you a daily summary of what matters.

The prompt (copy and paste this):

You are my Personal AI Assistant. Every morning at 8am, review my calendar, tasks, and any important updates. Then provide a brief, actionable summary covering:

  1. Top 3 priorities for today
  2. Any meetings that need preparation
  3. Follow-ups or replies I might be missing

Keep it short, clear, and actionable. Format as a bullet list.

That's it. Paste that into your agent's configuration, and every morning you'll wake up to a personalized briefing. No coding required.

Step 4: Workspace Organization

OpenClaw uses a workspace with personality files:

  • SOUL.md — Who the agent is (personality, values)
  • USER.md — Who you are (preferences, context)
  • MEMORY.md — Long-term memory
  • memory/ — Daily logs

Step 5: Model Selection

OpenClaw supports multiple models. Here's the cheat sheet:

Use Case Model Cost
Quick tasks, reminders GPT-4o Mini $0.15/1M tokens
Complex reasoning Sonnet 3.5 $3/1M tokens
Best quality GPT-4o $15/1M tokens

Pro tip: Start cheap, upgrade only when needed.

// TROUBLESHOOTING & DEBUGGING

Why Isn't My Agent Responding?

When your agent goes silent or behaves unexpectedly, here's the debugging checklist:

  1. Check the logs — Run openclaw status to see what's happening
  2. Verify API keys — Expired or invalid keys are the #1 cause of failures
  3. Test the model — Try a direct API call to verify connectivity
  4. Check memory — Corrupted memory files can cause weird behavior

Common Issues & Fixes

Agent Repeats Itself

Cause: Short context window or looping memory

Fix: Clear conversation history, reduce conversation turns, or switch to a model with larger context

Agent Ignores Instructions

Cause: Instructions buried in long prompts

Fix: Put core instructions at the start of the prompt. Use clear headings.

API Errors or Timeouts

Cause: Rate limits, network issues, or provider outages

Fix: Add retry logic, implement fallback models, check provider status pages

Memory Bloat

Cause: Memory files growing unbounded

Fix: Implement memory pruning. Archive old entries, keep only last 30 days in active memory

Debug Mode

Enable verbose logging to see exactly what your agent is thinking:

openclaw debug on
# Run your command
# Review detailed logs

Testing Prompts Before Production

  • Test with edge cases (empty input, bad data, unexpected formats)
  • Run 5-10 test conversations before deploying
  • Use a separate test channel before connecting to real messaging platforms
  • Log outputs for 24-48 hours to catch subtle issues

// MONITORING & HEALTHCHECKS

Your agents need supervision. Here's how to keep them honest.

Setting Up Healthchecks

OpenClaw supports cron-based healthchecks. Run these checks periodically:

# Add to your crontab
0 9 * * * /usr/local/bin/openclaw healthcheck --notify

What to Monitor

Metric Why It Matters Alert Threshold
Token usage Catch runaway spending > 2x daily average
Response time Agent may be stuck > 30 seconds
Error rate API or config issues > 5% of requests
Memory size Disk bloat, performance > 100MB

Alert Channels

Configure alerts to go where you'll actually see them:

  • Telegram/Signal — Fast, personal
  • Slack — Team visibility
  • Email — For critical failures only

Weekly Audit Routine

Set aside 15 minutes weekly to:

  1. Review token usage reports
  2. Check agent conversation logs for drift
  3. Verify integrations are still connected
  4. Archive or prune old memory files

Pro Tip: Shadow Mode

Run new agents in shadow mode — they suggest actions but don't execute them. Review their outputs for 1-2 weeks before enabling full autonomy.

// COST MODELING

Running AI agents costs money. Here's how to model and manage your spend.

Understanding Token Costs

Most providers charge per million tokens (input + output combined).

Model Input / 1M tokens Output / 1M tokens
GPT-4o Mini $0.15 $0.60
Claude 3.5 Sonnet $3.00 $15.00
GPT-4o $2.50 $10.00

Cost Estimates by Use Case

Use Case Daily Conversations Est. Monthly Cost
Daily briefing agent 10 $5-10
CS automation (50 accounts) 200 $30-50
Research agent (ongoing) 500+ $100-200

Cost Reduction Strategies

  • Use cheap models for simple tasks — Route reminders, lookups to Mini models
  • Implement caching — Don't re-fetch the same data
  • Limit conversation length — Truncate or summarize old context
  • Set budgets per agent — Cap spend before it gets out of hand
  • Monitor daily — Catch spikes early

Setting Up Budget Alerts

# In your config
{
  "budget": {
    "monthlyLimit": 50,
    "alertAt": 0.8,
    "notifyChannel": "telegram"
  }
}

// BUILDING CUSTOM SKILLS

Skills are reusable tool sets that extend your agent's capabilities. Here's how to build them.

What Is a Skill?

A skill is a collection of:

  • Tool definitions — What the agent can do
  • Implementation — The code that runs
  • Documentation — How to use it

Skill Structure

skills/
  └── my-skill/
      ├── SKILL.md        # Documentation
      ├── tools.js        # Tool definitions
      └── index.js        # Implementation

Example: Weather Skill

SKILL.md:

# Weather Skill

## Tools
- `get_weather` - Get current weather for a location
- `get_forecast` - Get 3-day forecast

## Usage
Ask: "What's the weather in [city]?"

tools.js:

module.exports = {
  tools: [
    {
      name: 'get_weather',
      description: 'Get current weather for a city',
      parameters: {
        city: { type: 'string', required: true }
      }
    }
  ]
};

Registering a Skill

// In your agent config
{
  "skills": ["weather", "calendar", "email", "my-custom-skill"]
}

When to Build Custom Skills

  • Repeated task across conversations
  • Custom API integrations
  • Domain-specific tools (e.g., CRM, support desk)
  • External service automation

Pro Tip: Start Simple

Don't over-engineer. One well-designed tool is better than a complex skill. Add more as needed.

// SECURITY MATTERS

Why Use a Dedicated Machine?

Running AI agents means giving them access to your digital life. A dedicated or freshly wiped machine keeps your personal data separate.

  • Isolation: If compromised, only the dedicated machine is affected
  • Clean slate: No old credentials, old accounts, or sensitive data
  • Peace of mind: You control exactly what's accessible

Essential Security Practices

  • Never put API keys in prompts — Use environment variables
  • Run locally — Your data stays on your machine
  • Review before sending — Don't let agents email externally unchecked
  • Use separate accounts — Create dedicated accounts for agent actions
  • Set usage alerts — Monitor token usage for unexpected spikes
  • Keep software updated — OpenClaw and your OS

What NOT to Do

  • Don't share your screen with agents running
  • Don't store passwords in plain text
  • Don't give agents admin access unless needed
  • Don't skip reviewing agent outputs
